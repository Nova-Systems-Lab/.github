# Contributing to Nova Systems Lab

Thank you for your interest in contributing to Nova Systems Lab.

We welcome contributions from developers, testers, designers, technical writers, students, and community members of all experience levels.

## Before Contributing

Please:

1. Read the project README and documentation.
2. Review the existing GitHub Discussions and Issues.
3. Check whether someone is already working on the same topic.
4. Read and follow our `CODE_OF_CONDUCT.md`.
5. Avoid submitting large changes without discussing them first.

For major features, architectural changes, or significant redesigns, open a GitHub Discussion before beginning implementation.

## Where to Start

You can contribute by:

- Reporting confirmed bugs
- Suggesting improvements
- Testing development builds
- Improving documentation
- Reviewing Pull Requests
- Writing or improving tests
- Contributing code
- Helping other community members
- Providing UI and UX feedback

Look for Issues labelled:

- `good first issue`
- `help wanted`
- `documentation`
- `testing`
- `bug`
- `enhancement`

Labels may vary between projects.

## Reporting Bugs

Before submitting a bug report:

- Search existing open and closed Issues.
- Confirm that the problem can be reproduced.
- Test the latest available version when possible.
- Remove private or sensitive information from logs.

A useful bug report should include:

- A clear and specific title
- The project and version affected
- Operating-system and environment details
- Steps to reproduce the problem
- Expected behaviour
- Actual behaviour
- Screenshots, logs, or error messages where relevant

Security vulnerabilities must not be reported publicly. Follow the instructions in `SECURITY.md`.

## Suggesting Features

Early ideas and brainstorming can begin in Discord.

Use GitHub Discussions for detailed proposals that should remain publicly searchable.

A good feature proposal should explain:

- What problem it solves
- Who would benefit from it
- How it could work
- Possible limitations or risks
- Any alternatives considered

A feature request does not guarantee implementation. Decisions depend on feasibility, security, maintainability, project scope, and available contributors.

## Development Workflow

Unless a repository documents a different process:

1. Fork the repository.
2. Create a branch from the appropriate development branch.
3. Make focused changes.
4. Add or update tests where relevant.
5. Update documentation when behaviour changes.
6. Commit with clear messages.
7. Push your branch.
8. Open a Pull Request.

Do not work directly on the stable branch.

Suggested branch names:

```text
feature/short-description
fix/short-description
docs/short-description
test/short-description
refactor/short-description
```

## Pull Request Guidelines

Pull Requests should:

- Focus on one logical change
- Have a clear title and description
- Explain why the change is needed
- Describe how the change was tested
- Link to a related Issue when possible
- Avoid unrelated formatting or refactoring
- Update documentation where necessary
- Pass required checks

Use one of the following in the Pull Request description when appropriate:

```text
Closes #123
Fixes #123
Resolves #123
```

Maintainers may request changes before merging.

## Code Quality

Contributions should aim to be:

- Clear
- Maintainable
- Secure
- Well documented
- Consistent with the existing architecture
- Limited to the necessary scope

Avoid:

- Unexplained copied code
- Large unrelated rewrites
- Hard-coded secrets
- Unsafe handling of user input
- Unnecessary dependencies
- Committing generated files unless required
- Breaking compatibility without discussion

## Testing

Where applicable:

- Test changes locally
- Add or update automated tests
- Confirm existing features still work
- Document manual testing steps
- Include relevant environment details

A change may be delayed or rejected if it cannot be tested safely.

## Documentation

Documentation contributions are welcome.

Please keep documentation:

- Clear and accurate
- Friendly to beginners
- Consistent with current project behaviour
- Free from unnecessary jargon
- Updated alongside related code changes
- Artificial Intelligence Tools

AI-assisted tools may be used, but contributors remain fully responsible for everything they submit.

You must:

- Review and understand generated code
- Test it properly
- Check licensing and attribution requirements
- Remove insecure or unnecessary code
- Be able to explain the contribution
- Never submit secrets, private data, or proprietary code to external tools

Unreviewed or unverifiable generated content may be rejected.

## Licensing

By submitting a contribution, you agree that it may be distributed under the licence used by the relevant repository.

Only submit work that you have the legal right to contribute.

Do not include proprietary, leaked, stolen, or licence-incompatible material.

## Communication

Use:

- Discord for quick discussion and coordination
- GitHub Discussions for detailed proposals
- GitHub Issues for actionable work
- Pull Requests for code and documentation changes

GitHub is the official source of truth for project decisions and development history.

## Review and Acceptance

Maintainers may accept, request changes to, postpone, or decline a contribution.

Possible reasons include:

- The change is outside project scope
- The implementation creates security or maintenance concerns
- The proposal needs further discussion
- The change duplicates existing work
- Tests or documentation are incomplete
- The contribution conflicts with project direction

Please remain respectful during review. Feedback is about the contribution, not the contributor.
